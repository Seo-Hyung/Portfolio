My role

함께 팀원과 함께 필요한 Page와 Component를 적절히 나눠 설계했습니다. Component는 Container Component와 Presentational Component를 나눠 Component별 역할을 뚜렷히 두었습니다. 그리고 공유할 Context를 함께 설계 후 구현했습니다.

개인 설계한 컴포넌트 구조대로 전체 틀을 짰습니다. 중복되는 레이아웃들은 모듈화하여 재사용할 수 있게 만들었습니다.

개인 로그인 / 회원가입

회원가입Form에 제어되는 컴포넌트를 이용해 비밀번호 확인란 일치 여부를 판단했습니다.

회원가입을 하면 새로운 token을 할당받고, 로그인하면 서버로부터 token을 받아와 localStorage에 저장해 사용자가 서비스를 이용할 수 있게 구현했습니다.

서버의 user list를 받아와 회원가입시 아이디 중복체크와 로그인 시 없는ID체크를 구현했습니다.

개인 영화 상세

영화 상세 정보를 서버에서 비동기 통신으로 받아와 화면에 그렸습니다.
줄거리에 포함된 다양한 문법들을 기준으로 개행해주어 사용자가 읽기 쉽게 구현했습니다.
많은 스틸컷들은 사용자가 직접 넘겨볼 수 있도록 캐러셀 라이브러리를 이용했습니다.

예매하기 버튼을 누르면 영화 id를 query String으로 넘겨서 해당 영화가 선택된 예약페이지로 바로 넘어가도록 구현했습니다.

개인 영화 예매

영화, 극장, 날짜를 선택하면 비동기 통신으로 서버에서 조건에 맞는 리스트를 받아와 띄우고, 세 가지를 모두 선택 시 시간을 선택할 수 있게 구현했습니다.

한 컴포넌트 내에서 서로 의존성을 가진 많은 상태들을 다루는 작업이었습니다.

모든 조건을 알맞게 선택하게 되면 Context에 data들을 담고, 좌석 페이지로 넘어가게 구현했습니다.

Issue
CORS 문제

로그인, 회원가입을 구현하면서 CORS 문제를 겪었습니다. Client에서 임시 방편으로 해결할 수 있는 방법들이 몇 가지 있었으나, 근본적인 해결법은 서버에서 해주어야 했습니다. 임시 해결법 중 하나인 Chrome확장 프로그램을 사용해 개발을 진행하다가 서버 개발자분들께 문제를 말씀드려 해결할 수 있었습니다. 비록 해결은 서버분들의 역할이었으나, 이 과정에서 CORS가 무엇인지, 왜 이러한 문제가 생기는 것인지 알게되었습니다.

한 페이지에서 여러 상태 공유 문제

예매 페이지를 구현할 때, 영화, 극장, 날짜, 시간, 좌석 등등의 많은 데이터가 공유되어야 했습니다. 전부 prop으로 해결하자니, 컴포넌트는 복잡해지고 코드는 길어져만 갔습니다. 이를 해결하기 위해 팀원과 의논해 Context를 만들어 사용자가 선택하는 예매 데이터들을 관리하고 공유했습니다.

이 중 영화, 극장, 날짜, 시간은 모두 한 화면에 출력되고, 서로 선택하는대로 조건에 맞는 리스트를 업데이트해주어야 했습니다. Container Component는 하나로 두지만, 각 리스트별로 Presentational Component를 쪼개어 컴포넌트 크기를 줄여 코드 흐름을 쉽게 파악할 수 있게 했습니다. 한 화면에서 선택에 따라 비동기 작업이 계속 이루어져야 했기 때문에 로딩인디케이터를 두어 사용자에게 시각적인 메세지를 주었습니다.
